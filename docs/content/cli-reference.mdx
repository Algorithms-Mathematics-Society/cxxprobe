# CLI Reference

```text
cxxprobe <subcommand> [OPTIONS] ...
```

cxxprobe has four subcommands: [`run`](#run) (sandbox a program directly),
[`new contest`](#new-contest--new-problem) / [`new problem`](#new-contest--new-problem)
(scaffolding), and [`test problem`](#test-problem) (judging). `-V`/`--version`
works at the top level without a subcommand.

## `run`

The original single-binary sandbox runner — everything cxxprobe did before it
grew contest/problem scaffolding. Everything after `program` is passed
through as that program's `argv` — put cxxprobe's own flags first.

```text
cxxprobe run [OPTIONS] program [args...]
```

| Flag | Short | Default | Description |
|---|---|---|---|
| `--memory-mb N` | `-m` | `256` | Memory limit in MiB |
| `--cpu DURATION` | `-t` | `5000ms` | CPU time limit |
| `--wall DURATION` | `-w` | `10000ms` | Wall-clock limit |
| `--pids N` | `-p` | `64` | Max process count |
| `--input FILE` | `-i` | (empty) | Stdin file; `-` reads from a pipe |
| `--expected FILE` | `-e` | (none) | Expected output — enables verdict judging |
| `--checker PATH` | | (none) | Custom checker binary (testlib ABI) |
| `--cases PATH` | | (none) | Batch mode: directory or manifest |
| `--json` | | off | Emit result as JSON |
| `--quiet` | `-q` | off | Suppress the metadata line (human mode) |
| `--no-color` | | off | Disable ANSI colors |

### Duration format

`--cpu` / `-t` and `--wall` / `-w` accept three forms:

```bash
--wall 2s       # 2 seconds
--wall 500ms    # 500 milliseconds
--wall 2000     # raw milliseconds (backward-compatible)
```

Only pure digit strings (optionally suffixed with `s` or `ms`) are accepted —
decimals, negative numbers, and other suffixes raise an error.

### Output

**Human mode (default)**: the sandboxed program's **stdout** is written to
cxxprobe's stdout, unmodified. Its **stderr** is written to cxxprobe's
stderr, unmodified. A metadata line is appended to stderr (suppress with
`-q`):

```text
[exit:0 | cpu:42ms | wall:47ms | mem:3.2MiB | AC]
```

The trailing verdict segment (` | AC`) only appears when judging is active
(`-e` or `--cases`).

**JSON mode (`--json`)**:

```json
{
  "exit_code": 0,
  "peak_memory_bytes": 3354624,
  "cpu_time_ms": 42,
  "wall_time_ms": 47,
  "verdict": "AC",
  "stdout": "15\n",
  "stderr": ""
}
```

The `verdict` key is only present when judging is active. Control characters
below `0x20` in `stdout`/`stderr` are escaped as `\uXXXX`.

### Verdicts

| Verdict | Meaning |
|---|---|
| `AC` | Accepted — output matches expected |
| `WA` | Wrong Answer |
| `TLE` | Time Limit Exceeded (wall or CPU) |
| `MLE` | Memory Limit Exceeded |
| `OLE` | Output Limit Exceeded (stdout ≥ 4 MiB) |
| `RE` | Runtime Error (non-zero exit) |

Priority when multiple conditions trigger simultaneously:
**TLE > MLE > OLE > RE > WA/AC**.

### Exit codes

<Callout type="info">
  Exit codes differ depending on whether judging (`-e` / `--cases`) is active.
</Callout>

**Single run, judging active:**
- `0` — AC
- `1` — any non-AC verdict (WA, TLE, MLE, OLE, RE)
- `2` — sandbox error (permission denied, binary not found, etc.)

**Single run, no judging:**
- Child's exit code, passed through (capped at 125)
- If killed by a signal: `128 + signal_number` (e.g. SIGKILL → `137`)
- `2` — sandbox error

**Batch mode (`--cases`):**
- `0` — all judged cases passed (or nothing was judged)
- `1` — at least one non-AC verdict
- `2` — at least one sandbox error

### Resource limits reference

| Resource | Flag | Default | Enforcement mechanism |
|---|---|---|---|
| Memory | `-m` / `--memory-mb` | 256 MiB | `memory.max` (cgroup v2) |
| Swap | (always 0) | 0 | `memory.swap.max=0` |
| CPU time | `-t` / `--cpu` | 5 s | `RLIMIT_CPU` (kernel-level kill) + `cpu.stat` (exact post-hoc verdict) |
| Wall clock | `-w` / `--wall` | 10 s | `timerfd` + `epoll` + `cgroup.kill` |
| PIDs | `-p` / `--pids` | 64 | `pids.max` (cgroup v2) |
| Stdout | (fixed) | 4 MiB | Drain thread discards past the cap |
| Stderr | (fixed) | 4 MiB | Drain thread discards past the cap |
| Stdin | (fixed) | 4 MiB | Capped when loading with `-i -` |

Both `RLIMIT_CPU` and the wall-clock timer are real kill mechanisms —
`RLIMIT_CPU` is a fast kernel-level backstop for CPU-bound runaways,
`timerfd`/`cgroup.kill` also catches non-CPU-bound cases (blocked I/O,
`sleep()`). `cpu.stat` remains the source of truth for the exact `TLE`
verdict regardless of which mechanism actually killed the process. See
[Sandbox Internals](/architecture/sandbox-internals) for the full mechanism.

## `new contest` / `new problem`

Scaffolds a contest or problem directory. Run from anywhere for `new
contest`; run from inside (or under) a contest directory for `new problem`.

```text
cxxprobe new contest "Name"
cxxprobe new problem "Name" [-C/--dir CONTEST_DIR]
```

| Flag | Description |
|---|---|
| `-C, --dir PATH` | Contest directory override (default: auto-detected by walking up from cwd looking for `contest.yaml`) |

See [Folder Structure](/architecture/folder-structure) for exactly what gets
generated and why, and [Contest Workflows](/guides/contest-workflows) for the
end-to-end authoring flow.

## `test problem`

Runs all three consolidated test types (manual `.in`/`.out`, symbolic source
checks, GTest-linked behavior checker) for one problem and reports an
aggregate verdict.

```text
cxxprobe test problem "Name" [-C/--dir CONTEST_DIR] [--submission PATH]
                              [--json] [-q/--quiet] [--no-color]
```

| Flag | Description |
|---|---|
| `-C, --dir PATH` | Contest directory override (same auto-detection as `new problem`) |
| `--submission PATH` | Grade an arbitrary source file instead of the problem's own `solution.cpp` — see [Contest Workflows](/guides/contest-workflows#grading-a-submission) |
| `--json` | Emit the full report as JSON |
| `-q, --quiet` | Suppress per-case/per-check detail (human mode) |
| `--no-color` | Disable ANSI colors |

`"Name"` resolves either as the problem's slug (its directory name) or its
exact `problem.yaml` `name:` field.

### Output shape

Each of the three sections gets an independent `status`:
`PASS | FAIL | SKIPPED | ERROR` — `SKIPPED` means that section is disabled in
`problem.yaml` (nothing to check); `ERROR` means an infra failure (e.g. a
compile error) as opposed to a genuine test failure.

```json
{
  "problem": "A: FileReader RAII",
  "slug": "a-filereader-raii",
  "submission": "solution.cpp",
  "overall": "FAIL",
  "tests": {
    "manual": {"status": "PASS", "passed": 8, "total": 8, "cases": ["..."]},
    "symbolic": {"status": "FAIL", "checks": ["..."]},
    "behavior": {"status": "PASS", "passed": 5, "total": 5, "cases": ["..."]}
  },
  "compile": {"solution": {"ok": true}, "behavior_binary": {"ok": true}}
}
```

### Exit codes

- `0` — overall `PASS` (or every enabled-and-relevant section trivially
  passed/was skipped)
- `1` — overall `FAIL` (at least one enabled section failed)
- `2` — overall `ERROR` (compile failure, missing problem, config error)
