# CLI Reference

```text
cxxprobe <subcommand> [OPTIONS] ...
```

cxxprobe has seven subcommands: [`identity`](#identity) (machine-readable build
provenance), [`run`](#run) (sandbox a program directly),
[`new contest`](#new-contest--new-problem) / [`new problem`](#new-contest--new-problem)
(scaffolding), [`test problem`](#test-problem) (judging),
[`bundle validate`](#bundle-validate) (deterministic bundle validation), and
[`serve`](#serve) (the HTTP judging service). `-V`/`--version` works at the
top level without a subcommand.

## `identity`

Prints the exact engine identity embedded at build time. The machine-readable
form is intended for deployment startup checks and durable validation
provenance:

```text
cxxprobe identity --json
```

It emits exactly one compact JSON object followed by a newline:

```json
{"contract":"cxxprobe.engine-identity","schema_version":1,"name":"cxxprobe","version":"0.9.0","commit":"0123456789abcdef0123456789abcdef01234567","dirty":false}
```

`commit` is either the full 40-character lowercase Git commit embedded by the
build or `unknown`. An unknown commit is always marked dirty. Production
consumers must compare every field with their pinned artifact identity and
reject `dirty=true`; this command does not inspect or guess repository state at
runtime. Without `--json`, the same values are printed as a human-readable
single line. The existing `-V`/`--version` output remains unchanged.

## `run`

The original single-binary sandbox runner — everything cxxprobe did before it
grew contest/problem scaffolding. Everything after `program` is passed
through as that program's `argv` — put cxxprobe's own flags first.

```text
cxxprobe run [OPTIONS] program [args...]
```

| Flag | Short | Default | Description |
|---|---|---|---|
| `--memory-mb N` | `-m` | `256` | Memory limit in MiB |
| `--cpu DURATION` | `-t` | `5000ms` | CPU time limit |
| `--wall DURATION` | `-w` | `10000ms` | Wall-clock limit |
| `--pids N` | `-p` | `64` | Max process count |
| `--input FILE` | `-i` | (empty) | Stdin file; `-` reads from a pipe |
| `--expected FILE` | `-e` | (none) | Expected output — enables verdict judging |
| `--checker PATH` | | (none) | Custom checker binary (testlib ABI) |
| `--cases PATH` | | (none) | Batch mode: directory or manifest |
| `--json` | | off | Emit result as JSON |
| `--quiet` | `-q` | off | Suppress the metadata line (human mode) |
| `--no-color` | | off | Disable ANSI colors |

### Duration format

`--cpu` / `-t` and `--wall` / `-w` accept three forms:

```bash
--wall 2s       # 2 seconds
--wall 500ms    # 500 milliseconds
--wall 2000     # raw milliseconds (backward-compatible)
```

Only pure digit strings (optionally suffixed with `s` or `ms`) are accepted —
decimals, negative numbers, and other suffixes raise an error.

### Output

**Human mode (default)**: the sandboxed program's **stdout** is written to
cxxprobe's stdout, unmodified. Its **stderr** is written to cxxprobe's
stderr, unmodified. A metadata line is appended to stderr (suppress with
`-q`):

```text
[exit:0 | cpu:42ms | wall:47ms | mem:3.2MiB | AC]
```

The trailing verdict segment (` | AC`) only appears when judging is active
(`-e` or `--cases`).

**JSON mode (`--json`)**:

```json
{
  "exit_code": 0,
  "peak_memory_bytes": 3354624,
  "cpu_time_ms": 42,
  "wall_time_ms": 47,
  "verdict": "AC",
  "stdout": "15\n",
  "stderr": ""
}
```

The `verdict` key is only present when judging is active. Control characters
below `0x20` in `stdout`/`stderr` are escaped as `\uXXXX`.

### Verdicts

| Verdict | Meaning |
|---|---|
| `AC` | Accepted — output matches expected |
| `WA` | Wrong Answer |
| `TLE` | Time Limit Exceeded (wall or CPU) |
| `MLE` | Memory Limit Exceeded |
| `OLE` | Output Limit Exceeded (stdout ≥ 4 MiB) |
| `RE` | Runtime Error (non-zero exit) |

Priority when multiple conditions trigger simultaneously:
**TLE > MLE > OLE > RE > WA/AC**.

### Exit codes

<Callout type="info">
  Exit codes differ depending on whether judging (`-e` / `--cases`) is active.
</Callout>

**Single run, judging active:**
- `0` — AC
- `1` — any non-AC verdict (WA, TLE, MLE, OLE, RE)
- `2` — sandbox error (permission denied, binary not found, etc.)

**Single run, no judging:**
- Child's exit code, passed through (capped at 125)
- If killed by a signal: `128 + signal_number` (e.g. SIGKILL → `137`)
- `2` — sandbox error

**Batch mode (`--cases`):**
- `0` — all judged cases passed (or nothing was judged)
- `1` — at least one non-AC verdict
- `2` — at least one sandbox error

### Resource limits reference

| Resource | Flag | Default | Enforcement mechanism |
|---|---|---|---|
| Memory | `-m` / `--memory-mb` | 256 MiB | `memory.max` (cgroup v2) |
| Swap | (always 0) | 0 | `memory.swap.max=0` |
| CPU time | `-t` / `--cpu` | 5 s | `RLIMIT_CPU` (kernel-level kill) + `cpu.stat` (exact post-hoc verdict) |
| Wall clock | `-w` / `--wall` | 10 s | `timerfd` + `epoll` + `cgroup.kill` |
| PIDs | `-p` / `--pids` | 64 | `pids.max` (cgroup v2) |
| Stdout | (fixed) | 4 MiB | Drain thread discards past the cap |
| Stderr | (fixed) | 4 MiB | Drain thread discards past the cap |
| Stdin | (fixed) | 4 MiB | Capped when loading with `-i -` |

Both `RLIMIT_CPU` and the wall-clock timer are real kill mechanisms —
`RLIMIT_CPU` is a fast kernel-level backstop for CPU-bound runaways,
`timerfd`/`cgroup.kill` also catches non-CPU-bound cases (blocked I/O,
`sleep()`). `cpu.stat` remains the source of truth for the exact `TLE`
verdict regardless of which mechanism actually killed the process. See
[Sandbox Internals](/architecture/sandbox-internals) for the full mechanism.

## `new contest` / `new problem`

Scaffolds a contest or problem directory. Run from anywhere for `new
contest`; run from inside (or under) a contest directory for `new problem`.

```text
cxxprobe new contest "Name"
cxxprobe new problem "Name" [-C/--dir CONTEST_DIR]
```

| Flag | Description |
|---|---|
| `-C, --dir PATH` | Contest directory override (default: auto-detected by walking up from cwd looking for `contest.yaml`) |

See [Folder Structure](/architecture/folder-structure) for exactly what gets
generated and why, and [Contest Workflows](/guides/contest-workflows) for the
end-to-end authoring flow.

## `bundle validate`

Validates one publishable question, materialized as a minimal contest tree,
and emits its immutable content identity. A bundle requires exactly one
direct-child problem directory; this keeps the digest question-scoped while
remaining directly usable with `cxxprobe test problem -C`.

```text
cxxprobe bundle validate CONTEST_DIR [--json]
```

The validator loads `contest.yaml` and every direct-child `problem.yaml`, then
recursively hashes every regular file. Paths are normalized to relative UTF-8
with `/` separators and sorted bytewise. Symlinks, special files, unsafe or
Windows-reserved names, ASCII case collisions, traversal references, symbolic
links, hard links, and trees over the conservative file/size/path limits are
rejected.

The bundle SHA-256 is the SHA-256 of the compact, ordered JSON manifest with
only its `bundle_sha256` self-digest field omitted. `problem.yaml` version 1
continues to emit schema 1 byte-for-byte, including its established digest.
Version 2 emits schema 2 and adds the canonical `public` object described
below. Both schemas include sorted file paths, sizes, per-file SHA-256 values,
and the problem's resolved `execution.compiler` (`cxx`, `std_flag`, `flags`,
and `extra_sources`) and `execution.limits` (`memory_bytes`, `cpu_time_ms`,
`wall_time_ms`, and `max_pids`). Resolution uses the same cxxprobe problem
defaults and overrides as judging, so effective execution and public-projection
changes are part of problem identity. The identity deliberately excludes
archive/compression metadata, mtimes, ownership, and permission bits.
Referenced statement, solution, checker, and enabled test resources must exist
inside the problem directory. `--json` emits the ordered manifest. A
publishable question must enable at least one manual, symbolic, or behavior
check. Successful validation exits `0`; invalid input exits `2`.

### Public projection (schema 2)

Candidate visibility is an explicit allowlist in `problem.yaml` version 2:

```yaml
version: 2
statement: problem.md
public:
  statement: true
  assets:
    - path: public/diagram.png
      media_type: image/png
  starter:
    path: public/starter.cpp
    language: cpp
```

If `public` or one of its fields is omitted, the corresponding value is
private. `statement: true` exposes only the configured statement. Assets and
the optional starter must be under `public/`; assets are limited to bounded
PNG, JPEG, or WebP files whose lowercase extension, declared media type, and
file signature agree. Statement and starter files must be bounded UTF-8 text,
and the starter must be a lowercase `.cpp` file with language `cpp`.
Execution resources are always private: the reference solution, compiler extra
sources, test directory/manifest/checker, behavior checker, and
`problem.yaml` cannot be assigned a public role. The manifest uses
problem-relative paths prefixed by the slug and sorts assets bytewise:

```json
{
  "public": {
    "statement": "a-problem/problem.md",
    "assets": [
      {"path": "a-problem/public/diagram.png", "media_type": "image/png"}
    ],
    "starter": {"path": "a-problem/public/starter.cpp", "language": "cpp"}
  }
}
```

An entirely private v2 projection is always serialized as `statement: null`,
`assets: []`, and `starter: null`. A v1 file that defines `public` is rejected,
including `public: null`, so an older manifest cannot accidentally opt in.

Each problem entry in the JSON manifest exposes the exact effective execution
configuration that is covered by `bundle_sha256`:

```json
{
  "slug": "a-problem",
  "name": "A Problem",
  "execution": {
    "compiler": {
      "cxx": "g++",
      "std_flag": "c++23",
      "flags": ["-O2", "-Wall"],
      "extra_sources": []
    },
    "limits": {
      "memory_bytes": 268435456,
      "cpu_time_ms": 5000,
      "wall_time_ms": 10000,
      "max_pids": 64
    }
  }
}
```

## `test problem`

Runs all three consolidated test types (manual `.in`/`.out`, symbolic source
checks, GTest-linked behavior checker) for one problem and reports an
aggregate verdict.

```text
cxxprobe test problem "Name" [-C/--dir CONTEST_DIR] [--submission PATH]
                              [--json] [-q/--quiet] [--no-color]
```

| Flag | Description |
|---|---|
| `-C, --dir PATH` | Contest directory override (same auto-detection as `new problem`) |
| `--submission PATH` | Grade an arbitrary source file instead of the problem's own `solution.cpp` — see [Contest Workflows](/guides/contest-workflows#grading-a-submission) |
| `--json` | Emit the full report as JSON |
| `-q, --quiet` | Suppress per-case/per-check detail (human mode) |
| `--no-color` | Disable ANSI colors |

`"Name"` resolves either as the problem's slug (its directory name) or its
exact `problem.yaml` `name:` field.

### Output shape

Each of the three sections gets an independent `status`:
`PASS | FAIL | SKIPPED | ERROR` — `SKIPPED` means that section is disabled in
`problem.yaml` (nothing to check); `ERROR` means an infra failure (e.g. a
compile error) as opposed to a genuine test failure.

```json
{
  "problem": "A: FileReader RAII",
  "slug": "a-filereader-raii",
  "submission": "solution.cpp",
  "overall": "FAIL",
  "tests": {
    "manual": {"status": "PASS", "passed": 8, "total": 8, "cases": ["..."]},
    "symbolic": {"status": "FAIL", "checks": ["..."]},
    "behavior": {"status": "PASS", "passed": 5, "total": 5, "cases": ["..."]}
  },
  "compile": {"solution": {"ok": true}, "behavior_binary": {"ok": true}}
}
```

### Exit codes

- `0` — overall `PASS` (or every enabled-and-relevant section trivially
  passed/was skipped)
- `1` — overall `FAIL` (at least one enabled section failed)
- `2` — overall `ERROR` (compile failure, missing problem, config error)

## `serve`

Starts the HTTP judging service — the same `problem::load_from_dir` →
`judge::run_problem` pipeline `test problem` uses, exposed over REST + SSE
for a contest platform (or your own scripts) to submit against instead of
shelling out to the CLI. See [Running the HTTP Judging
Service](/guides/http-judging-service) for a full walkthrough and
[architecture](/architecture/http-service) for how it's built.

```text
cxxprobe serve [OPTIONS]
```

| Flag | Default | Description |
|---|---|---|
| `-C, --contest-dir PATH` | auto-detect via `contest.yaml` | Contest directory to serve problems from |
| `--bind ADDRESS` | `0.0.0.0` | Address to bind the API to |
| `--port PORT` | `8191` | API port |
| `--ui` | off | Also serve the embedded developer UI |
| `--ui-port PORT` | `8181` | Developer UI port (only with `--ui`) |
| `--workers N` | `4` | Judging worker threads |
| `--queue-capacity N` | `256` | Max queued submissions before `POST /submissions` returns `503` |
| `--http-threads N` | `4` | HTTP connection-handling threads |
| `--db PATH` | `cxxprobe-serve.sqlite3` | SQLite database file for submission state |

### Endpoints

| Method | Path | Description |
|---|---|---|
| `GET` | `/health` | Worker/queue status |
| `GET` | `/metrics` | Prometheus text exposition (or JSON with `Accept: application/json`) |
| `GET` | `/problems` | List problems in the served contest |
| `GET` | `/problems/{slug}` | Statement, limits, language — never the reference solution |
| `POST` | `/submissions` | Submit `{problem_slug, language, source}` — returns `202` immediately |
| `GET` | `/submissions/{id}` | Status + full `JudgeReport` once finished |
| `GET` | `/submissions` | Recent submission history (`?limit=N`, default 50, max 200) |
| `GET` | `/events` | SSE stream of submission/worker lifecycle events (`?submission_id=` to scope) |

No authentication — `cxxprobe serve` assumes it's reachable only from a
trusted caller. Errors are `{"error": "...", "message": "..."}` with `400`
(bad request), `404` (unknown problem/submission), or `503` (queue full,
`Retry-After` set).

Graceful shutdown on `SIGINT`/`SIGTERM`: stops accepting new connections,
lets any in-flight judging finish (a worker is never killed mid-judge), then
exits.
