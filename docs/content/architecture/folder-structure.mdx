# Folder Structure

`cxxprobe new contest` and `cxxprobe new problem` generate this structure —
it's not just a convention, `cxxprobe test problem` expects it. The
lower-level `cxxprobe run --cases` (see [Batch Mode](/guides/batch-mode))
still works against any ad hoc directory of `.in`/`.ans` files if you're not
using the scaffolding at all.

## A contest

```bash
cxxprobe new contest "AMS Ascent"
```

<FileTree>
  <FileTree.Folder name="ams-ascent" defaultOpen>
    <FileTree.File name="contest.yaml" />
  </FileTree.Folder>
</FileTree>

`contest.yaml` is a small marker file (`version`, `name`, `description`) —
`cxxprobe new problem` and `cxxprobe test problem` walk up from the current
directory looking for it to find the contest root (or pass `-C/--dir`
explicitly). Problem directories sit flat as direct children — there's no
extra `problems/` nesting level and no tool-enforced ordering; if you want
letter/number ordering visible in the problem list, put it in the title you
pass to `new problem` (e.g. `"A: FileReader RAII"`) — cxxprobe never
interprets that prefix, it just becomes part of the slug.

## A problem

```bash
cd ams-ascent
cxxprobe new problem "A: FileReader RAII"
```

<FileTree>
  <FileTree.Folder name="a-filereader-raii" defaultOpen>
    <FileTree.File name="problem.yaml" />
    <FileTree.File name="problem.md" />
    <FileTree.File name="solution_template.cpp" />
    <FileTree.File name="checker_gtest.cpp" />
    <FileTree.Folder name="tests" />
  </FileTree.Folder>
</FileTree>

| File / folder | Purpose |
|---|---|
| `problem.yaml` | Version-2 config: private-by-default public projection, compiler/limits overrides, and the 3 consolidated test type settings — see [CLI Reference](/cli-reference#public-projection-schema-2) |
| `problem.md` | Human-readable statement; candidate-visible only when `public.statement: true` |
| `solution_template.cpp` | Copy to `solution.cpp` and fill in — the reference solution |
| `checker_gtest.cpp` | GTest `TEST()` cases exercising the submission directly (consolidated type 3) |
| `tests/` | `.in`/`.ans` pairs (consolidated type 1) — empty until you add cases |

The directory name (`a-filereader-raii`) is a mechanical slug of the title
you passed — lowercase, non-alphanumeric runs collapsed to `-`. It's what
you pass to `cxxprobe test problem`, though the exact `problem.yaml` `name:`
also resolves (so both `cxxprobe test problem a-filereader-raii` and
`cxxprobe test problem "A: FileReader RAII"` work).

A freshly scaffolded, unfilled problem is a clean baseline `PASS` — manual
tests and symbolic checks are inferred **disabled** until you actually add
`tests/*.in` files or `symbolic.must_include`/`must_not_include` entries;
the placeholder `TEST()` in `checker_gtest.cpp` passes trivially, so the
pipeline itself is verified working before you've written a line of the
actual solution.

The scaffold declares an empty `public` allowlist. To publish raster assets or
a starter, create paths such as `public/diagram.png` and
`public/starter.cpp`, then list them explicitly in `public.assets` or
`public.starter`. Merely placing a file in `public/` does not expose it.

<Callout type="warning">
  `checker_gtest.cpp` includes the submission via the `CXXPROBE_SOLUTION_FILE`
  macro cxxprobe defines at compile time — don't replace that with a
  hardcoded `#include "solution.cpp"`, or [`--submission`
  grading](/guides/contest-workflows#grading-a-submission) will silently test
  the wrong file. The submission's own `main()` is renamed out of the way
  (`#define main solution_main`) so it doesn't collide with GTest's `main()`.
</Callout>

## Manual tests: directory or manifest

`tests/` holds `<label>.in` / `<label>.ans` pairs (`.out` also accepted for
the answer):

<FileTree>
  <FileTree.Folder name="tests" defaultOpen>
    <FileTree.File name="1.in" />
    <FileTree.File name="1.ans" />
    <FileTree.File name="2.in" />
    <FileTree.File name="2.ans" />
    <FileTree.File name="10.in" />
    <FileTree.File name="10.ans" />
  </FileTree.Folder>
</FileTree>

Labels sort **numerically** (`10.in` after `2.in`, not before it as plain
string sort would); a case without a matching answer file still runs, just
unjudged. If cases don't map cleanly to files on disk, point `problem.yaml`'s
`tests.manifest` at a YAML/JSON manifest instead of `tests.dir` — see [Batch
Mode](/guides/batch-mode#yamljson-manifest) for the manifest schema (the
mechanism is identical, just addressed from `problem.yaml` instead of
`--cases`).

    <FileTree.Folder name="b-graph-diameter">
      <FileTree.File name="problem.yaml" />
      <FileTree.File name="solution.cpp" />
    </FileTree.Folder>
  </FileTree.Folder>
</FileTree>

Each problem is self-contained and independently gradable — `cxxprobe test
problem` resolves one problem directory at a time. There's no shared global
config to keep in sync.

<Callout type="info">
  Per-problem isolation matches how cxxprobe itself runs each case: every
  invocation gets its own cgroup leaf, so nothing about a problem's test data
  or a contestant's submission is shared state between runs.
</Callout>

## Contestant submissions

`cxxprobe test problem --submission <path>` grades an arbitrary source file
through all 3 consolidated checks in place of the problem's own
`solution.cpp` — see [Contest
Workflows](/guides/contest-workflows#grading-a-submission) for the full
grading-loop pattern over many contestants:

<FileTree>
  <FileTree.Folder name="submissions" defaultOpen>
    <FileTree.Folder name="team-01">
      <FileTree.File name="solution.cpp" />
    </FileTree.Folder>
    <FileTree.Folder name="team-02">
      <FileTree.File name="solution.cpp" />
    </FileTree.Folder>
  </FileTree.Folder>
</FileTree>

```bash
for sub in submissions/*/solution.cpp; do
    cxxprobe test problem a-filereader-raii --submission "$sub" --json
done
```

If you're not using the scaffolded checks at all and just want raw
`.in`/`.ans` judging against a pre-built binary per contestant, the
lower-level `cxxprobe run --cases` loop from [Batch
Mode](/guides/batch-mode) still works identically.
