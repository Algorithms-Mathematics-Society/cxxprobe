# Folder Structure

cxxprobe doesn't impose a rigid project layout — `--cases` just needs a folder
of paired files or a manifest, and everything else (checker, generator,
solutions) is a plain path you pass on the command line. This page documents
the conventions that work well in practice, used throughout the
[guides](/guides/contest-workflows).

## A single problem

The minimum cxxprobe needs is a directory of paired `.in`/`.ans` files:

<FileTree>
  <FileTree.Folder name="tests" defaultOpen>
    <FileTree.File name="1.in" />
    <FileTree.File name="1.ans" />
    <FileTree.File name="2.in" />
    <FileTree.File name="2.ans" />
    <FileTree.File name="10.in" />
    <FileTree.File name="10.ans" />
  </FileTree.Folder>
</FileTree>

- Each case is a `<label>.in` / `<label>.ans` pair sharing a stem. `.out` is
  also accepted as the answer extension.
- Labels are sorted **numerically**, not lexicographically — `10.in` sorts
  after `2.in`.
- A case without a matching answer file still runs, just unjudged.

```bash
cxxprobe --cases tests/ ./solution
```

A fuller problem directory, once you add a reference solution, checker, and
generator, looks like:

<FileTree>
  <FileTree.Folder name="problemA" defaultOpen>
    <FileTree.File name="statement.md" />
    <FileTree.Folder name="tests" defaultOpen>
      <FileTree.File name="1.in" />
      <FileTree.File name="1.ans" />
      <FileTree.File name="2.in" />
      <FileTree.File name="2.ans" />
    </FileTree.Folder>
    <FileTree.File name="solution.cpp" />
    <FileTree.File name="checker.cpp" />
    <FileTree.File name="gen.cpp" />
  </FileTree.Folder>
</FileTree>

| File / folder | Purpose | Used with |
|---|---|---|
| `tests/` | Test data (`.in`/`.ans` pairs) | `--cases problemA/tests/` |
| `solution.cpp` | Reference solution, compiled before grading | passed as `program` |
| `checker.cpp` | Optional testlib-ABI checker, compiled once | `--checker ./checker` |
| `gen.cpp` | Optional test-case generator | driving a [stress-test loop](/guides/stress-testing) |
| `statement.md` | Human-readable problem statement | not read by cxxprobe |

None of these names are enforced by the tool — `solution`, `checker` and `gen`
are conventional filenames, compiled to plain binaries and referenced by
whatever path you give the CLI.

## A manifest instead of a directory

If cases don't map cleanly to files on disk (inline data, custom per-case
labels, cases assembled by another tool), replace `tests/` with a manifest:

<FileTree>
  <FileTree.Folder name="problemA" defaultOpen>
    <FileTree.File name="cases.yaml" />
    <FileTree.File name="solution.cpp" />
  </FileTree.Folder>
</FileTree>

```yaml filename="cases.yaml"
- input: "tests/1.in"
  answer: "tests/1.ans"
- input_data: "3\n1 2 3\n"
  answer_data: "6\n"
  label: "inline-3"
```

Paths inside the manifest are resolved relative to the manifest file, so a
manifest and a `tests/` directory can coexist — see [Batch
Mode](/guides/batch-mode#yamljson-manifest) for the full schema.

## A multi-problem contest

Nothing changes at contest scale — it's just one problem directory per
problem, each self-contained and independently gradable:

<FileTree>
  <FileTree.Folder name="contest" defaultOpen>
    <FileTree.Folder name="problemA" defaultOpen>
      <FileTree.File name="tests/" />
      <FileTree.File name="solution.cpp" />
      <FileTree.File name="checker.cpp" />
    </FileTree.Folder>
    <FileTree.Folder name="problemB">
      <FileTree.File name="tests/" />
      <FileTree.File name="solution.cpp" />
    </FileTree.Folder>
    <FileTree.Folder name="problemC">
      <FileTree.File name="…" />
    </FileTree.Folder>
  </FileTree.Folder>
</FileTree>

A grading script just iterates the problem directories and, for each,
resolves that problem's own `tests/` and `checker` — there's no shared global
config to keep in sync. See [Contest
Workflows](/guides/contest-workflows#grading-many-submissions) for a working
grading-loop example.

<Callout type="info">
  Per-problem isolation matches how cxxprobe itself runs each case: every
  invocation gets its own cgroup leaf, so nothing about a problem's test data
  or a contestant's submission is shared state between runs.
</Callout>

## Contestant submissions

Grading scripts commonly expect one binary (or one source file to compile)
per contestant, named consistently so a loop can glob over them:

<FileTree>
  <FileTree.Folder name="submissions" defaultOpen>
    <FileTree.Folder name="team-01">
      <FileTree.File name="solution" />
    </FileTree.Folder>
    <FileTree.Folder name="team-02">
      <FileTree.File name="solution" />
    </FileTree.Folder>
  </FileTree.Folder>
</FileTree>

```bash
for sub in submissions/*/solution; do
    cxxprobe --cases problemA/tests/ --checker ./problemA/checker --json "$sub"
done
```

This is exactly the layout used in the [grading-many-submissions
example](/guides/contest-workflows#grading-many-submissions).
