# cxxprobe

**cxxprobe** is a sandboxed evaluation engine for competitive programming and
technical interviews. It runs an untrusted binary inside a locked-down Linux
sandbox (cgroup v2 + a user namespace), enforces memory, CPU, wall-clock and
process-count limits, captures stdout/stderr, and — optionally — judges the
result against expected output.

It ships as a single static-ish CLI, `cxxprobe`, plus a small C++ library
(`libcxxprobe`) that the CLI is built on, so you can embed sandboxed execution
directly into your own judge or contest platform.

```bash
cxxprobe run -m 256 -w 2s -t 1s -i tests/1.in -e tests/1.ans ./solution
```

```text
15
[exit:0 | cpu:42ms | wall:47ms | mem:3.2MiB | AC]
```

<Callout type="info">
  cxxprobe requires **Linux with cgroup v2**. It runs fine as an unprivileged
  user once a delegated cgroup subtree is set up — see [Getting
  Started](/getting-started).
</Callout>

## Why cxxprobe

- **Real resource isolation.** Memory, swap, CPU time, wall-clock time and PID
  count are enforced by the kernel via cgroup v2 — not by best-effort polling.
- **No root required.** Sandboxing uses `CLONE_NEWUSER | CLONE_NEWNS`, so it
  works under an unprivileged user once cgroup delegation is configured once.
- **Judge built in.** Token-based diffing out of the box, or bring your own
  [testlib-ABI checker](/guides/custom-checker) for floating point / multiple
  valid answers / interactive problems.
- **Three ways to judge a submission, run together.** Manual `.in`/`.out`
  cases, comment/string-aware source checks (`must_include`/
  `must_not_include` — "did they actually use `std::bit_cast`, not
  `memcpy`?"), and a GTest-linked behavior checker that tests the
  submission's actual implementation, not just its stdin/stdout. `cxxprobe
  test problem` runs all three and reports one aggregate verdict.
- **Contest scaffolding.** `cxxprobe new contest` / `cxxprobe new problem`
  generate the whole directory structure — `problem.yaml`, a statement
  template, a solution template, a checker template — so there's nothing to
  hand-roll.
- **Machine-readable output.** `--json` emits exit code, resource usage,
  verdict and captured output as JSON for grading pipelines and CI.

## Where a contest lives

`cxxprobe new contest "Name"` and `cxxprobe new problem "Name"` scaffold a
contest directory and its problems — see [Folder

and [Contest Workflows](/guides/contest-workflows) for the end-to-end
authoring and grading flow.
