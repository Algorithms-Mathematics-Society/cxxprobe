# cxxprobe

**Set up a coding contest in three commands, and grade every submission with
confidence — not just "did the output match," but "did they actually solve
it the way the problem asked."**

```bash
curl -sSf https://algorithms-mathematics-society.github.io/cxxprobe/install.sh | sh
```

```bash
cxxprobe new contest "AMS Ascent"
cxxprobe new problem "A: FileReader RAII"
# fill in solution.cpp, then:
cxxprobe test problem a-filereader-raii
```

```text
Problem: A: FileReader RAII (a-filereader-raii)

Manual tests    PASS (3/3)
Symbolic checks PASS
Behavior tests  PASS (4/4)

---
Overall: PASS
```

That's a whole problem — scaffolded, authored, and judged three different
ways at once — without writing any grading infrastructure yourself. Every
run happens inside a real sandbox, so it's safe to point this at code you
don't trust yet.

## Why cxxprobe

- **Three ways to judge, together — pick what fits each problem.** A plain
  "does stdin/stdout match" problem needs nothing extra. A problem that's
  really testing a technique ("use `std::bit_cast`, not `memcpy`") gets a
  one-line source check. A problem where correctness can't be seen from
  stdout at all — RAII cleanup, move semantics, exception safety — gets a
  real behavior checker. Combine any of the three per problem; `cxxprobe
  test problem` reports one clear verdict either way.
- **Scaffolding, not folder conventions to memorize.** `cxxprobe new
  contest` / `cxxprobe new problem` generate everything — statement,
  solution template, test directory, checker template — so there's nothing
  to hand-roll and nothing to get subtly wrong.
- **Grade contestants, not just your own reference solution.** `cxxprobe
  test problem --submission path/to/contestant.cpp` runs the exact same
  three-way judging against anyone's code — that's the whole grading
  pipeline, one command per submission.
- **Real safety, not best-effort.** Every run — including compiling the
  submission — happens in a sandbox where memory, CPU time, wall-clock time,
  and process count are enforced by the Linux kernel itself. A fork bomb or
  an infinite loop in a contestant's code can't take down your grading
  host, and it costs you nothing to set up beyond a one-time step — no root
  required.
- **Built for pipelines, not just terminals.** Everything speaks `--json` —
  exit code, resource usage, verdict, and full per-check detail — so
  grading a hundred submissions in CI is exactly as easy as grading one by
  hand.

<Callout type="info">
  cxxprobe needs **Linux with cgroup v2** (WSL2 works too) and one quick
  one-time setup step — see [Getting Started](/getting-started). No root
  required after that.
</Callout>

## Prefer the low-level primitive?

Everything above is built on top of `cxxprobe run` — a single sandboxed
execution with explicit limits, no contest scaffolding required:

```bash
cxxprobe run -m 256 -w 2s -t 1s -i tests/1.in -e tests/1.ans ./solution
```

```text
15
[exit:0 | cpu:42ms | wall:47ms | mem:3.2MiB | AC]
```

Reach for this directly if you're integrating cxxprobe into your own
existing tooling rather than using its contest scaffolding — see [Single-Run
Judging](/guides/single-run-judging) and [Batch Mode](/guides/batch-mode).
