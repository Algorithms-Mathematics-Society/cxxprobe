# Stress Testing & Debugging

Hand-written tests rarely catch every edge case. The standard competitive
programming technique is: generate a random small input, run two solutions
(a fast one under test and a slow-but-obviously-correct brute force) against
it, and compare. cxxprobe's single-run mode is a natural fit for this loop
since it's cheap to invoke thousands of times and keeps every run isolated.

## Generate + judge in a pipeline

```bash
for i in $(seq 1 100); do
    ./gen "$i" > tmp.in
    ./brute < tmp.in > tmp.ans
    result=$(cxxprobe -i tmp.in -e tmp.ans --checker ./checker --json ./solution)
    verdict=$(jq -r '.verdict' <<< "$result")
    if [ "$verdict" != "AC" ]; then
        echo "FAIL on seed $i: $verdict"
        cat tmp.in
        break
    fi
done
```

Each iteration re-derives the expected answer from the brute force, so you
don't need pre-baked test data — just a generator and a reference
implementation you trust.

## Stress test two solutions directly

If you don't have a checker yet, or you're just diffing two implementations'
raw output:

```bash
for i in $(seq 1 1000); do
    ./gen "$i" > tmp.in
    out_a=$(cxxprobe -i tmp.in -q ./sol_a)
    out_b=$(cxxprobe -i tmp.in -q ./sol_b)
    if [ "$out_a" != "$out_b" ]; then
        echo "Mismatch on seed $i"
        cat tmp.in
        break
    fi
done
```

`-q` suppresses the metadata line so `out_a`/`out_b` are pure stdout, safe to
compare directly.

<Callout type="info">
  Keep generated inputs **small** (tens to low hundreds of elements) for stress
  testing — the goal is to explore edge cases fast across thousands of seeds,
  not to test performance. Use dedicated large inputs separately for TLE/MLE
  checks.
</Callout>

## Debugging a specific failure

Once you have a failing seed, re-run it without `-q` to see resource usage,
and drop `--checker`/`-e` to see raw output for manual inspection:

```bash
./gen 42 > tmp.in
cxxprobe -i tmp.in ./solution
```

```text
<raw stdout>
[exit:0 | cpu:12ms | wall:14ms | mem:1.8MiB]
```

If the process was killed by a signal, the exit code in JSON mode is
negative — e.g. `-11` is SIGSEGV, `-9` is SIGKILL (from a resource limit).
See [Exit Codes](/cli-reference#exit-codes).

## Checking memory usage in isolation

```bash
cxxprobe --json -i big_input.txt ./solution \
  | jq '{"mem_mib": (.peak_memory_bytes/1048576|.*10|round/10), "cpu_ms": .cpu_time_ms}'
```

Useful for tuning a problem's `-m`/`-t` limits against the reference solution
before publishing test data — see [Contest
Workflows](/guides/contest-workflows#setting-a-problem).
