# Contest Workflows

This page walks through using cxxprobe as the evaluation engine for an actual
contest or problem set — from scaffolding to grading a batch of contestant
submissions. For the on-disk layout referenced below, see [Folder
Structure](/architecture/folder-structure).

## Scaffold the contest and a problem

<Steps>

### Create the contest

```bash
cxxprobe new contest "AMS Ascent"
cd ams-ascent
```

### Create a problem

```bash
cxxprobe new problem "A: FileReader RAII"
cd a-filereader-raii
```

This generates `problem.yaml`, `problem.md`, `solution_template.cpp`,
`checker_gtest.cpp`, and an empty `tests/`.

</Steps>

## Setting a problem

<Steps>

### Write the reference solution

```bash
cp solution_template.cpp solution.cpp
```

Fill it in.

### Add manual test cases (consolidated type 1)

Hand-write small tests (edge cases, examples from the statement) as
`tests/1.in`/`tests/1.ans`, `tests/2.in`/`tests/2.ans`, etc. — same
`.in`/`.ans` format as [Batch Mode](/guides/batch-mode). No `problem.yaml`
change needed; `tests.enabled` is inferred true once `tests/` has data.

### Add symbolic requirements, if the problem needs them (consolidated type 2)

For problems that are really testing whether a specific technique was used
— "must use `std::bit_cast`, not `memcpy`" — add requirements to
`problem.yaml`:

```yaml filename="problem.yaml"
symbolic:
  must_include: ["std::bit_cast"]
  must_not_include:
    - pattern: "\\bmemcpy\\s*\\("
      regex: true
      message: "memcpy for type punning is UB — use std::bit_cast"
```

The scan strips comments and string/char literals first, so a pattern
appearing only in a comment or a string doesn't false-positive. Most
problems leave this section empty — it's for the subset where "did they use
the right tool" matters and can't be observed from stdin/stdout alone. See
[FAQ](/faq#why-regex-based-symbolic-checks-and-not-a-real-astlibclang) for
why this is regex-based rather than a full AST parse.

### Add behavior tests, if stdin/stdout isn't enough (consolidated type 3)

For problems where the interesting behavior isn't visible through a
`main()` reading stdin — RAII cleanup timing, move-only types, exception
safety — write real `TEST()` cases in `checker_gtest.cpp` against the
submission's actual interface:

```cpp filename="checker_gtest.cpp"
#include <gtest/gtest.h>

#define main solution_main
#include CXXPROBE_SOLUTION_FILE
#undef main

TEST(FileReader, ClosesOnDestruction) {
    // exercise the submission's classes/functions directly
}
```

### Validate the reference solution

```bash
cxxprobe test problem a-filereader-raii
```

```text
Problem: A: FileReader RAII (a-filereader-raii)

Manual tests    PASS (3/3)
Symbolic checks PASS
Behavior tests  PASS (2/2)

---
Overall: PASS
```

If it's not a clean `PASS`, either the solution or the test data is wrong —
fix it before publishing the problem.

### Stress-test with a generator (optional but recommended)

Add randomized large cases with a generator and re-validate against a
brute-force solution — see [Stress Testing &
Debugging](/guides/stress-testing).

### Fix the limits for the problem

Once you know the reference solution's typical resource usage, override
`problem.yaml`'s `limits:` with headroom (2-3× the reference solution's time
is a common rule of thumb) — this is what every submission gets graded
under, including via `--submission`.

</Steps>

## Grading a submission

`--submission` runs the exact same three consolidated checks against an
arbitrary source file instead of the problem's own `solution.cpp` — this is
how you actually grade a contestant, not just validate your own reference
solution:

```bash
cxxprobe test problem a-filereader-raii --submission contestant.cpp --json > result.json
jq '.overall, .tests.manual.passed, .tests.manual.total' result.json
```

```json
"FAIL"
6
8
```

Per-section, per-case detail (which manual case failed with what verdict,
which symbolic check was unmet, which named `TEST()` failed) is in
`.tests.manual.cases[]` / `.tests.symbolic.checks[]` /
`.tests.behavior.cases[]` — see [CLI Reference](/cli-reference#test-problem)
for the full JSON shape.

## Grading many submissions

```bash
#!/usr/bin/env bash
set -euo pipefail

for sub in submissions/*/solution.cpp; do
    team=$(basename "$(dirname "$sub")")
    result=$(cxxprobe test problem a-filereader-raii --submission "$sub" --json)
    overall=$(jq -r '.overall' <<< "$result")
    echo "$team: $overall"
    echo "$result" > "results/$team.json"
done
```

Each `cxxprobe` invocation — compile, manual tests, symbolic scan, behavior
checker — is fully isolated (its own cgroup leaf, its own namespace) per
submission, so submissions can't interfere with each other even if run
concurrently — parallelize the loop with `xargs -P` or GNU `parallel` if you
have many submissions and need throughput.

<Callout type="info">
  Because every run is sandboxed independently — including the **compile
  step** (a compile-time bomb like runaway template recursion is bounded by
  its own generous default limits) — a misbehaving or malicious submission
  can't affect the grading host or other submissions running alongside it.
</Callout>

## Regression-testing your own judge

If you're building a platform around cxxprobe, treat the reference solution
as a regression test: re-run `cxxprobe test problem` against it (no
`--submission`) in CI whenever test data, symbolic requirements, or the
behavior checker changes, and fail the build if it stops being an overall
`PASS`. This catches a broken problem before contestants do.

## When you don't need the scaffolding at all

Everything above builds on the lower-level `cxxprobe run` primitive — a
directory of `.in`/`.ans` files and a pre-built binary, no `problem.yaml`
required. If you already have your own contest tooling and just want
sandboxed judging, use `run --cases` directly:

```bash
cxxprobe run -m 256 -w 2s --cases tests/ ./solution
```

See [Batch Mode](/guides/batch-mode) for that path in full.
