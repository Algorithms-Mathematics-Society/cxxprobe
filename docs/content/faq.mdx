# FAQ

### `sandbox error: create cgroup root /sys/fs/cgroup/cxxprobe: Permission denied`

Your user hasn't been delegated a writable cgroup subtree yet. Run the
one-time setup from [Getting
Started](/getting-started#one-time-cgroup-delegation):

```bash
echo "+memory +cpu +pids" | sudo tee /sys/fs/cgroup/cgroup.subtree_control
sudo mkdir -p /sys/fs/cgroup/cxxprobe
sudo chown -R "$(whoami):$(whoami)" /sys/fs/cgroup/cxxprobe
```

This resets on every reboot (`/sys/fs/cgroup` is a virtual filesystem) —
re-run it, or wire it into a startup script.

### Do I need to run cxxprobe as root?

No. Root works (and is what CI uses, since it already owns all of
`/sys/fs/cgroup`), but an unprivileged user works equally well once the
one-time cgroup delegation above has been done. Sandboxing itself uses a user
namespace (`CLONE_NEWUSER`), which is specifically designed to not require
host privilege.

### Does cxxprobe work on WSL2?

Yes — WSL2 mounts cgroup v2 by default with the `nsdelegate` option, and
delegation works the same as native Linux. One internal test
(`CgroupTest.AddPidCgroupProcsContainsPid`) self-skips under `nsdelegate`
because direct `cgroup.procs` migration from outside the ancestor chain is
blocked by the kernel — this is expected and doesn't affect actual sandboxed
runs, which have an `RLIMIT_AS` fallback for exactly this case.

### Does cxxprobe work on macOS or Windows (natively)?

No. cxxprobe is Linux-only — it depends directly on cgroup v2 and Linux user
namespaces, neither of which exist on macOS or native Windows. Use WSL2 on
Windows.

### Is the sandbox a real security boundary?

It's resource and process-tree isolation — memory/CPU/wall-clock/PID limits
are enforced by the kernel, and the sandboxed process can't see the host
process tree. It is **not** a full container runtime: there's no seccomp
filter, no network namespace, and no filesystem overlay, so it's appropriate
for trusted-but-buggy or resource-abusive code (contest submissions, student
code), not as a boundary against a determined adversary with local filesystem
access. See [Sandbox Internals](/architecture/sandbox-internals#process-tree-isolation)
for the full picture.

### Why did I get `TLE` when the program clearly finished fast?

Two different things can trigger `TLE`: the **wall-clock** timer killing the
process, or **CPU time** exceeding `-t` — enforced live via `RLIMIT_CPU`
(usually the faster of the two for a CPU-bound runaway) and always double
-checked post-hoc against `cpu.stat` for the exact verdict. If your process
legitimately used more CPU than `-t` allows — even briefly, even if it
returned before the `-w` wall-clock deadline — it's still `TLE`. See
[Resource Limits](/architecture/resource-limits#cpu-time).

### What does exit code `137` mean?

`137 = 128 + 9`, i.e. the process was killed by signal 9 (`SIGKILL`) — usually
because it hit the wall-clock or memory limit. See [Exit
Codes](/cli-reference#exit-codes) for the full `128 + signal` convention.

### My checker isn't being found

`--checker` (under `run`) resolves the path **as given** — absolute or
relative to the current working directory, never via `$PATH`. Use
`./checker`, not `checker`. Also remember `--checker` must be combined with
`--expected` or `--cases`; it errors out (exit `2`) on its own. See [Custom
Checkers](/guides/custom-checker). Under `test problem`, the equivalent is
`problem.yaml`'s `tests.checker`, resolved relative to the problem
directory.

### Can output limits (4 MiB) be configured?

No — the 4 MiB stdout/stderr/stdin caps are fixed, not exposed as flags. See
[Resource Limits](/architecture/resource-limits#output-caps).

### What's the difference between `--expected` and `--cases`?

Both are `run` flags. `-e`/`--expected` judges a **single** run against one
expected-output file. `--cases` runs the **same binary** against many test
cases (a directory or a YAML/JSON manifest) in one invocation, printing a
per-case verdict and a summary. See [Single-Run
Judging](/guides/single-run-judging) vs. [Batch Mode](/guides/batch-mode).
`test problem`'s manual-tests section is `--cases`-equivalent judging driven
from `problem.yaml` instead of flags.

### What does `cxxprobe new problem` actually scaffold, and why is nothing enabled by default?

`problem.yaml`, `problem.md`, `solution_template.cpp`, `checker_gtest.cpp`,
and an empty `tests/` — see [Folder
Structure](/architecture/folder-structure#a-problem) for the full list.
Manual tests and symbolic checks start **disabled by inference**: `tests/`
has no `.in` files yet, and `symbolic.must_include`/`must_not_include` are
empty lists. Only the behavior checker starts active, because its
placeholder `TEST()` passes trivially — so a freshly scaffolded, completely
unfilled-in problem is a clean baseline `PASS`, proving the pipeline itself
works before you've written any real content. Add `tests/*.in` files or
`symbolic:` entries to activate those sections.

### How is `checker_gtest.cpp` compiled against my solution?

cxxprobe compiles `checker_gtest.cpp` as a single translation unit — it
`#include`s the submission itself via `CXXPROBE_SOLUTION_FILE` (a macro
cxxprobe defines at compile time pointing at whichever file is actually
being graded — `solution.cpp` by default, or whatever `--submission` was
passed). The submission's own `main()` is renamed out of the way with
`#define main solution_main` so it doesn't collide with GTest's own `main()`
(linked in via `-lgtest_main`). One compiler invocation, sandboxed like any
other run — see [Contest Workflows](/guides/contest-workflows#setting-a-problem).

### Why regex-based symbolic checks and not a real AST/libclang?

Comment/string-aware regex is enough to answer "was this identifier/pattern
used or not" without pulling in libclang, which isn't cleanly
Conan-installable and would be a much larger dependency for what's usually a
simple keyword-usage question. It's not a full parse — it can't tell you
whether `std::bit_cast` was used *correctly*, only whether the text is
present outside a comment or string literal. If that precision ever proves
insufficient in practice, an AST-based backend is on the project's backlog
(see `PLAN.md`), not ruled out — just not built yet.

### Can I use cxxprobe as a library instead of the CLI?

Yes. Beyond the original
[`cxxprobe::sandbox::run()`](https://github.com/Algorithms-Mathematics-Society/cxxprobe/blob/main/include/cxxprobe/sandbox.hpp)
(argv + stdin + `Limits` → exit status/resource usage/captured output),
`libcxxprobe` also exposes the whole contest/problem pipeline as library
calls: `cxxprobe::problem::load_from_dir()` parses `problem.yaml`, and
`cxxprobe::judge::run_problem()` runs all 3 consolidated checks and returns
a `JudgeReport` struct — the same thing `cxxprobe test problem --json`
prints, but as native C++ types your own judge platform can call directly
instead of shelling out to the CLI and re-parsing its stdout.

### Does `cxxprobe serve` have authentication?

No, not in v1 — it assumes it's only reachable from a trusted caller (a
contest platform, an internal network), and focuses purely on judging. A
future `ApiKeyMiddleware` would slot into the existing middleware chain
without touching anything else if that's ever needed. See [HTTP Judging
Service architecture](/architecture/http-service#whats-deliberately-out-of-scope-v1).

### Is `cxxprobe serve --ui` a contest platform?

No — it's a small embedded developer UI for local testing and debugging
(think Prometheus + Prometheus UI, or MinIO + MinIO Console), not a
replacement for a real contest platform. No users, no auth, no scoreboards.
It talks to the API through the exact same public endpoints any other
client would use — see [Running the HTTP Judging
Service](/guides/http-judging-service#the-optional-developer-ui).

### Why is `cxxprobe serve --ui` one binary with no Node.js involved?

The UI is vanilla JS + [Alpine.js](https://alpinejs.dev/) +
[htmx](https://htmx.org/) + [CodeMirror 5](https://codemirror.net/5/),
vendored as plain committed static files — no bundler, no build step, ever.
A small Python script embeds every asset file into the C++ binary at build
time as a byte array + lookup table, so there's no runtime filesystem
dependency and nothing extra to install or ship alongside `cxxprobe-cli`.
